PHASE 2 — PYTHON CONNECTION TO SITL
────────────────────────────────────────────────────────────────────────────────

GOAL: Connect Python to the simulated drone and read/send data.

ATTEMPT 1 — DroneKit:
  Installed dronekit and connected to tcp:127.0.0.1:5762.
  Problem: DroneKit error —
    "class Parameters(collections.MutableMapping)" — Python 3.10 removed
    collections.MutableMapping, moved to collections.abc.MutableMapping.
  Solution: Manually patched dronekit __init__.py:
    Changed: class Parameters(collections.MutableMapping, HasObservers)
    To:      class Parameters(collections.abc.MutableMapping, HasObservers)
  Result: DroneKit connected successfully, battery/altitude/mode readable.

ATTEMPT 2 — Switching to PyMAVLink:
  DroneKit is outdated and incompatible with EKF type 10 (needed for
  episodic resets). Switched to PyMAVLink (pymavlink library) which is
  lower level and more reliable.
  
  Key discovery: PyMAVLink by default returns STALE queued messages.
  Problem: After several episodes, altitude readings were delayed — showing
    old values because the message queue was full.
  Solution: Wrote get_fresh_msg() function that drains the queue first,
    always returning the latest message:
      while True:
          new_msg = master.recv_match(type=type, blocking=False)
          if new_msg is None: break
          msg = new_msg
      if msg is None:
          msg = master.recv_match(type=type, blocking=True, timeout=2)

  Also requested all data streams at 10Hz on connection:
    master.mav.request_data_stream_send(..., MAV_DATA_STREAM_ALL, 10, 1)

OUTCOME: Reliable PyMAVLink connection reading fresh position, attitude,
         battery data in real time. ✅

PHASE 3 — CORE DRONE CONTROL FUNCTIONS
────────────────────────────────────────────────────────────────────────────────

GOAL: Build all core Python functions needed to control the drone.

FUNCTIONS BUILT AND TESTED:

  1. set_guided()
     Sets ArduCopter to GUIDED mode (custom_mode = 4).
     Required before any autonomous commands.
     Problem first time: Drone not moving after arm — was in STABILIZE mode.
     Solution: Always call set_guided() before arm().

  2. arm()
     Sends MAV_CMD_COMPONENT_ARM_DISARM with param1=1.
     Polls HEARTBEAT to confirm armed state.
     Problem: First run after SITL start, thrust loss check blocked arming.
     Solution: Changed to force arm using magic number 21196:
       MAV_CMD_COMPONENT_ARM_DISARM, param1=1, param2=21196

  3. force_disarm()
     Sends MAV_CMD_COMPONENT_ARM_DISARM with param2=21196.
     Works even if drone is in unstable state, mid-air, or crashed.
     Tested: Armed drone at 10m altitude, force disarmed — drone dropped
     immediately. Confirmed safe emergency stop.

  4. takeoff(alt)
     Sends MAV_CMD_NAV_TAKEOFF.
     Problem 1: Fixed altitude threshold (alt * 0.95) failed after teleport
       resets because ground level changed (EKF10 reports -0.8m as ground).
     Solution: Relative altitude tracking:
       ground_alt = get_alt() before takeoff
       rose = current_alt - ground_alt
       if rose >= alt * 0.90: break
     This works regardless of what EKF10 reports as ground level.

  5. get_pos()
     Reads LOCAL_POSITION_NED message.
     Returns (north, east, alt, vx, vy, vz) in up-positive convention.
     Note: NED uses z-down, so alt = -msg.z and vz = -msg.vz.

  6. get_attitude()
     Reads ATTITUDE message.
     Returns (roll, pitch, yaw) in radians.

  7. get_battery()
     Reads BATTERY_STATUS message.
     Returns (battery_remaining %, current_consumed mAh).
     Note: Voltage in SITL is not linked to current_consumed — they are
     separate calculations. Only current_consumed is reliable for tracking
     energy efficiency.

  8. send_velocity(vx, vy)
     Sends SET_POSITION_TARGET_LOCAL_NED with velocity-only bitmask.
     vx = north velocity, vy = east velocity, vz always 0 (altitude held).
     Bitmask: 0b0000111111000111 — velocity bits active only.

  9. teleport_home()
     Triggers Lua reset script via MAVLink parameter SIM_ERES_RESET=1.
     Lua script then calls sim:set_pose() to instantly teleport drone.

OUTCOME: All core functions working reliably. ✅

GOAL: Instantly teleport drone back to start position between RL episodes.

ATTEMPT 1 — Bruno's full Lua scaffolding:
  Cloned GSoC 2025 repo: github.com/b-andreoni/GSoC

  Problem 1: Script disables GPS (GPS_TYPE=0) and uses EKF type 10. This
    broke DroneKit connection entirely.
  Problem 2: Script controlled drone movement itself (state 1 = cruise to
    target), conflicting with Python PPO control.
  Problem 3: TARGET_THRESH_M = 0.01 (1cm) — drone could never reach target,
    never reset.
  Problem 4: Yaw rotated 45 degrees each reset causing EKF confusion.
  Problem 5: Battery not resetting between episodes.
  Decision: Keep only sim:set_pose() from Bruno's work. Rebuild everything
    else in Python.

ATTEMPT 2 — Custom minimal Lua script (reset_only.lua):
  Built a minimal Lua script that ONLY handles teleport reset.
  Python controls everything else (arm, takeoff, fly, disarm).
  
  Lua script structure:
    - Saves home location ONCE at startup (home_saved flag)
    - Watches SIM_ERES_RESET parameter
    - When Python sets SIM_ERES_RESET=1:
        sim:set_pose() to home_location
        arming:disarm()
        battery:reset_remaining(0, 100.0)
        param:set("SIM_BATT_VOLTAGE", 12.6)
  
  Problem: sim:set_pose() was teleporting drone visually but LOCAL_POSITION_NED
    still showed old coordinates (e.g., North:101.1 after teleport).
  
  Root cause: Bug in ArduPilot's SIM_Aircraft.cpp. The set_pose() function
    only updated the NED offset, not the global position origin:
      aircraft.position = aircraft.home.get_distance_NED_double(loc);
    This means coordinates are relative to old origin, not new position.
  
  Solution: Applied Bruno's fix from GitHub PR #29616:
    Changed SIM_Aircraft.cpp line 1434:
    FROM: aircraft.position = aircraft.home.get_distance_NED_double(loc);
    TO:   aircraft.home     = loc;
          aircraft.origin   = loc;
          aircraft.position = Vector3d(0, 0, 0);
    Rebuilt ArduPilot: ./waf copter
  
  Result: After fix, teleport correctly resets North=0, East=0, Alt=0. ✅
  
  Additional problem: H icon (home) in Mission Planner moved to drone's
    current position after each flight, causing teleport to go to wrong place.
  Solution: Save home location using ahrs:get_location() ONCE at startup
    with home_saved=true flag. Never update again.

BATTERY RESET:
  Problem: battery:reset_remaining_mavlink(0, 100) — wrong function name.
  Solution: battery:reset_remaining(0, 100.0) — correct Lua binding.
  Also: SIM_BATT_VOLTAGE set to 12.6 on each reset to prevent voltage drop.
  Also: SIM_BATT_CAP_AH set to 200 for large capacity buffer.

OUTCOME: Full episodic reset working — position, attitude, velocity, battery
         all reset correctly on each episode. ✅

PHASE 5 — FAILSAFE PROBLEMS DURING TRAINING
────────────────────────────────────────────────────────────────────────────────

PROBLEM: Training kept stopping after ~2 iterations due to various failsafes.

FAILSAFE 1 — Battery voltage failsafe:
  Voltage dropped below threshold triggering FAILSAFE mode.
  Solution: Disabled all battery failsafes in Lua at startup:
    param:set("BATT_FS_LOW_ACT", 0)
    param:set("BATT_FS_CRT_ACT", 0)
    param:set("BATT_LOW_VOLT", 0)
    param:set("BATT_CRT_VOLT", 0)
    param:set("BATT_ARM_VOLT", 0)

FAILSAFE 2 — Potential thrust loss (1, 3, 4):
  After many teleport resets, motor simulation got confused.
  MOT_THST_HOVER value drifted due to in-flight learning.
  Solution:
    param:set("MOT_THST_HOVER", 0.35)  -- fixed value
    param:set("MOT_HOVER_LEARN", 0)    -- stop hover learning

FAILSAFE 3 — EKF lane switch:
  EKF kept switching between IMU0 and IMU1 after resets.
  Solution:
    param:set("EK3_AFFINITY", 0)
    param:set("EK3_IMU_MASK", 1)  -- use only IMU0

FAILSAFE 4 — Dead reckoning + vibration failsafe:
  FS_DR_ENABLE triggered when EKF lost position after resets.
  FS_VIBE_ENABLE triggered from fake vibrations during teleport.
  Solution:
    param:set("FS_DR_ENABLE", 0)
    param:set("FS_VIBE_ENABLE", 0)
    param:set("FS_EKF_ACTION", 0)
    param:set("FS_GCS_ENABLE", 0)
    param:set("FS_THR_ENABLE", 0)
    param:set("FS_CRASH_CHECK", 0)

KEY LESSON: All these params must be set at SITL STARTUP (top of Lua script),
  not just during reset. First episode runs before any reset is triggered.

OUTCOME: Training runs continuously without failsafe interruptions. ✅

PHASE 6 — GYMNASIUM ENVIRONMENT (DroneEnv.py)
────────────────────────────────────────────────────────────────────────────────

GOAL: Build a complete Gymnasium-compatible RL environment.

OBSERVATION SPACE (23 floats):
  [0]  north          — NED north position from origin (m)
  [1]  east           — NED east position from origin (m)
  [2]  alt            — altitude above ground, up-positive (m)
  [3]  dist_north     — target_north - north (m)
  [4]  dist_east      — target_east - east (m)
  [5]  dist           — Euclidean distance to target (m)
  [6]  vx             — north velocity (m/s)
  [7]  vy             — east velocity (m/s)
  [8]  vz             — vertical velocity, up-positive (m/s)
  [9]  roll           — roll angle (rad)
  [10] pitch          — pitch angle (rad)
  [11] yaw            — yaw angle (rad)
  [12] heading_error  — signed angle from yaw to target bearing (rad)
  [13] battery_pct    — battery remaining (%)
  [14] ep_mah         — mAh consumed since episode start
  [15-22]             — obstacle distances N/NE/E/SE/S/SW/W/NW (placeholder 999.0)

  DESIGN DECISION: Obstacle distance slots included NOW as placeholders.
  The neural network input layer is fixed at creation. Adding obs later
  requires full retraining. With placeholders, real rangefinder data can
  be connected later with only fine-tuning needed — no architecture change.

ACTION SPACE (2 floats):
  [vx, vy] — horizontal velocity commands in m/s, range [-5, 5]
  Altitude is fixed at 10m, held by ArduPilot's PID controller.
  vz is observed but not commanded — intentional design.

REWARD FUNCTION:
  +150   Target reached (dist < 5m)
  -100   Out of bounds (|north| or |east| > 250m)
  -100   Crash detected (alt < 1m after step 20)
  -50    Battery critical (battery_pct < 5%)
  -20..0 Timeout (proximity partial credit — closer = less penalty)
  -0.05  Per-step time penalty (pressure to finish quickly)
  ±5.0   Distance progress (approach/retreat, clipped to ±5)
  ±1.5   Heading alignment (cos(heading_error) × 1.5)
  -*     Battery efficiency (ep_mah × 0.001)

  Heading error reward: drone gets positive reward when facing target,
  negative when facing away — encourages efficient straight-line navigation.

  Timeout partial credit: instead of flat -50 on timeout, reward is
  proportional to how close drone got — encourages progress even if
  target not reached.

RESET SEQUENCE:
  1. teleport_home() — triggers Lua sim:set_pose()
  2. Randomize TARGET_NORTH, TARGET_EAST in [-100, 100]
     Reject if both < 20m (too close to origin)
  3. set_guided() → arm() → takeoff(10m)
  4. Snapshot start_mah for per-episode energy tracking

EPISODE TERMINATION:
  - Target reached (dist < 5m) → +150 reward
  - Out of bounds → -100 reward
  - Crash (alt < 1m) → -100 reward
  - Battery dead → -50 reward
  - Timeout (600 steps) → proximity partial credit

OUTCOME: Complete Gymnasium environment built and verified. ✅

PHASE 8 — KEY TECHNICAL DISCOVERIES
────────────────────────────────────────────────────────────────────────────────

1. ArduPilot PR #29616 — sim:set_pose() NED offset bug:
   Bug: sim:set_pose() only updated NED offset, not global origin.
   Effect: After teleport, LOCAL_POSITION_NED still showed old coordinates.
   Fix: Modified SIM_Aircraft.cpp line 1434 to reset origin and position.
   Status: Bruno submitted PR #29616 — not yet merged into main ArduPilot.
   Action: Verified fix works for Python RL training. Should comment on PR.

2. EKF type 10 is essential for episodic resets:
   Nate confirmed: AHRS_EKF_TYPE=10 is a special SITL condition that enables
   state resets via sim:set_pose(). Cannot be removed.
   Impact: DroneKit fails with EKF10 (needs GPS). PyMAVLink works fine.

3. Stale message queue problem:
   PyMAVLink queues messages. During heavy training, queue builds up and
   recv_match returns old data. Solved with get_fresh_msg() function that
   drains queue before returning latest message.

4. NED coordinate convention:
   MAVLink uses NED (North-East-Down) — z is positive downward.
   All altitude and vz values negated in get_pos() for intuitive up-positive.

5. Battery tracking approach:
   SITL voltage and current_consumed are NOT linked.
   Voltage drops are simulation artifacts, not physics.
   Only current_consumed (mAh) is reliable for energy efficiency reward.
   Per-episode consumption tracked as: ep_mah = current_consumed - start_mah

────────────────────────────────────────────────────────────────────────────────
PROBLEM 1 — HOME POINT DRIFTING ON INSTANCE 1
────────────────────────────────────────────────────────────────────────────────

SYMPTOM:
  Instance 1's home icon in MavProxy/Mission Planner moved to wherever the
  drone crashed at the end of each episode, instead of staying fixed.
  This caused the next teleport to send the drone to the WRONG place.
  Instance 0 was mostly fine; Instance 1 was always broken.

ROOT CAUSE — Two bugs compounding together:

  BUG A: Blind time.sleep(1.5) after teleport trigger.
    teleport_home() sends SIM_ERES_RESET=1 via MAVLink param_set then Python
    sleeps 1.5 seconds assuming Lua is done. For Instance 1, sim:set_pose() +
    EKF convergence sometimes takes longer than 1.5 s, so the next command
    (set_home) fires while the AHRS still reports the old position.

  BUG B: MAV_CMD_DO_SET_HOME sent with param1=1 ("use current position").
    After teleport_home() + sleep, Python sent:
      MAV_CMD_DO_SET_HOME, param1=1
    param1=1 means "set home to my CURRENT position". Because Bug A caused
    the EKF to still be at the old position, ArduPilot read back the stale
    coordinates and stored those as the new home. The Lua script had correctly
    called ahrs:set_home(home_location) but Python immediately overwrote it
    with the wrong stale position.

SOLUTION:
  1. Replace blind sleep with polling: added _wait_for_lua_teleport() which
     polls SIM_ERES_STATE until Lua sets it to 1, confirming teleport done.
  2. Changed MAV_CMD_DO_SET_HOME to param1=0 with EXPLICIT lat/lon/alt:
       HOME_LAT = -35.3632610
       HOME_LON = 149.1652300
       HOME_ALT = 584.0
     These are decoded from the Lua script's hardcoded location values.
     param1=0 means "use the coordinates in params 5/6/7", never current pos.
     This fires correctly regardless of what the EKF currently reports.
  3. Added _set_home_explicit() function in Python to encapsulate this.

OUTCOME: Home point now stays fixed for both instances. ✅

────────────────────────────────────────────────────────────────────────────────
PROBLEM 2 — LUA SCRIPT BOOT ERROR: "No parameter: SYSID_THISMAV"
────────────────────────────────────────────────────────────────────────────────

SYMPTOM:
  After --wipe + SCR_ENABLE=1 + reboot, the SITL MAVProxy console showed:
    Lua: ./scripts/reset_only.lua:15: No parameter: SYSID_THISMAV
  The error repeated every few seconds. The Lua loop() never started.
  No teleport, no failsafe disabling — script completely dead.

ROOT CAUSE:
  Line 15 of reset_only.lua was:
    local SYS_ID_PARAM = Parameter("SYSID_THISMAV")
  Parameter() at the TOP LEVEL of a Lua script runs immediately when the
  script loads — before ArduPilot's parameter table is fully registered.
  SYSID_THISMAV does not exist yet at that moment, so Parameter() throws.
  The entire script stopped at line 15. Nothing below it ever ran.

  The variable SYS_ID_PARAM was also NEVER USED anywhere in the script —
  it was dead code. The update() function already correctly reads the same
  value via param:get("SYSID_THISMAV") which is inside a function and only
  called after boot is complete.

SOLUTION:
  Deleted the two dead-code lines:
    -- Create a pointer to SYSID_THISMAV (More reliable than param:get)
    local SYS_ID_PARAM = Parameter("SYSID_THISMAV")
  No other changes needed. param:get("SYSID_THISMAV") inside update() still
  works correctly because it runs at loop time, not at script load time.

OUTCOME: Script loads cleanly on both instances, no boot errors. ✅

────────────────────────────────────────────────────────────────────────────────
PROBLEM 3 — TELEPORT DETECTION ALWAYS TIMING OUT (PARAM_VALUE APPROACH)
────────────────────────────────────────────────────────────────────────────────

SYMPTOM:
  After the Lua boot fix, _wait_for_lua_teleport() (which polled
  SIM_ERES_STATE via param_request_read_send + recv_match PARAM_VALUE)
  always printed:
    [InstanceX] WARNING: Lua teleport confirmation timed out!
  Despite the 6-second timeout, it never detected SIM_ERES_STATE = 1.

ROOT CAUSE:
  ArduPilot continuously emits PARAM_VALUE messages in the background as
  part of its own parameter synchronisation with MavProxy. When Python
  called recv_match(type='PARAM_VALUE'), it grabbed the FIRST PARAM_VALUE
  that arrived — almost always a completely different parameter from the
  background flood — and discarded it. The actual SIM_ERES_STATE response
  was buried and never matched within the timeout window.

FIRST FIX ATTEMPT — STATUSTEXT detection:
  Changed _wait_for_lua_teleport() to listen for STATUSTEXT "Teleport
  Success" instead of PARAM_VALUE. The Lua script sends this message at the
  end of its teleport block. STATUSTEXT has its own isolated message type
  with no background flood, so matching it should be reliable.

  Problem: This also timed out every time.

  Root cause of STATUSTEXT failure:
    pymavlink's recv_match(type=X) reads messages from the TCP socket and
    DISCARDS any message that doesn't match the type filter — they are NOT
    buffered. While force_disarm() loops on recv_match(type='HEARTBEAT'),
    it reads and throws away every non-HEARTBEAT message that arrives,
    including the STATUSTEXT "Teleport Success" sent by Lua. By the time
    _wait_for_lua_teleport() started listening, the STATUSTEXT was gone.

FINAL FIX — Fixed sleep with race condition removed:
  The message-based detection approaches were fundamentally unreliable
  because pymavlink has no message buffer — discarded messages are lost.
  Solution: Replace all detection logic with time.sleep(2.0).
    - Lua's loop runs every 50ms.
    - 2 seconds = 40 loop iterations of margin.
    - Teleport is guaranteed complete before Python proceeds.
  Also removed _flush_statustext() and all polling code.

OUTCOME: Simple and reliable. No more timeout warnings. ✅

────────────────────────────────────────────────────────────────────────────────
PROBLEM 4 — RACE CONDITION: LUA DISARMS DRONE MID-TAKEOFF
────────────────────────────────────────────────────────────────────────────────

SYMPTOM:
  The Python output showed:
    After takeoff [Instance0] : -0.24m
    After takeoff [Instance1] : -0.07m
  Altitude after 8 seconds of "takeoff" was essentially 0 — drone never rose.
  User observed: "the drone is just armed, brought to an alt, and disarmed,
  repeated again and again."

ROOT CAUSE — Race condition between Python arm and Lua disarm:
  The Lua script contained:
    if arming:is_armed() then arming:disarm() end
  This was intended to guarantee a clean state on teleport. But when the
  teleport detection timed out (Problem 3), Python proceeded immediately
  to set_guided() → arm() → takeoff(). Lua then fired LATE (after Python
  had already armed and the drone was climbing) and called arming:disarm(),
  killing the drone mid-takeoff. The drone fell back to ground before the
  8-second takeoff sleep finished, giving ~0m altitude.

  The race timeline:
    Python: teleport_home() → timeout (6-7s) → arm() → takeoff() ← climbing
    Lua:    (fires late, sees armed=True)  →  arming:disarm()  ← kills drone

  The Lua disarm was redundant to begin with: Python's force_disarm() +
  while self.is_armed(): pass already guarantees the drone is disarmed
  before SIM_ERES_RESET=1 is ever sent. Lua can never see an armed drone
  at the moment it processes the trigger.

SOLUTION:
  Removed the arming:disarm() line from reset_only.lua entirely.
  Added a comment explaining why it was removed and why it's safe.
  With time.sleep(2.0) guaranteeing Lua finishes before Python arms,
  and Lua no longer able to disarm, the race condition is fully closed.

OUTCOME: Drone arms, takes off, and flies without being killed. ✅

────────────────────────────────────────────────────────────────────────────────
PROBLEM 5 — PHYSICAL TELEPORT NOT HAPPENING (sim:set_pose SILENT FAILURE)
────────────────────────────────────────────────────────────────────────────────

SYMPTOM:
  After all the above fixes, training ran without errors or warnings.
  Home stayed fixed. Drone armed and flew. But the drone did NOT physically
  return to the home location between episodes — it armed and took off from
  wherever it had crashed, accumulating drift across episodes.

ROOT CAUSE — Two bugs in the Lua sim:set_pose() call:

  BUG A: Wrong instance index for Instance 1.
    The Lua script computed:
      local instance_idx = 0
      if sys_id > 1 then instance_idx = math.floor(sys_id - 1) end
      sim:set_pose(instance_idx, home_location, ...)
    For Instance 1 (SYSID_THISMAV=2), instance_idx=1.
    BUT each SITL instance is a completely separate ArduPilot process with
    its own physics engine. Index 0 ALWAYS means "myself" within that
    engine. Instance 1's Lua calling sim:set_pose(1, ...) asks its own
    simulator to move a vehicle at index 1, which does not exist → silent
    failure, no teleport, no error message.
    Instance 0 called sim:set_pose(0, ...) which is correct and should work.

  BUG B: Uninitialised Quaternion.
    The Lua script called:
      sim:set_pose(0, home_location, Quaternion(), ...)
    Quaternion() in ArduPilot Lua creates a new object but does NOT
    initialise its fields to the identity quaternion {w=1, x=0, y=0, z=0}.
    The fields start at zero ({w=0, x=0, y=0, z=0}), which is not a valid
    rotation. Passing a garbage quaternion to sim:set_pose() causes
    ArduPilot to either reject the pose or apply a corrupted rotation,
    resulting in silent failure for both instances.

SOLUTION:
  BUG A fix: Always pass 0 as the instance index regardless of sys_id:
    sim:set_pose(0, home_location, q, zero_v, zero_v)
    The instance_idx variable is kept only for the gcs:send_text() log msg.

  BUG B fix: Initialise the quaternion with from_euler():
    local q = Quaternion()
    q:from_euler(0, 0, 0)   -- roll=0, pitch=0, yaw=0 → level, facing north
    This produces a valid identity-equivalent rotation.

  ADDITIONAL HARDENING: Call sim:set_pose() three times in quick succession.
    One call can occasionally be absorbed by the physics scheduler tick
    without taking effect. Three consecutive calls guarantee at least one
    lands on a clean scheduler frame:
      sim:set_pose(0, home_location, q, zero_v, zero_v)
      sim:set_pose(0, home_location, q, zero_v, zero_v)
      sim:set_pose(0, home_location, q, zero_v, zero_v)

  Python side: Added _set_ekf_origin() which sends MAVLink
    SET_GPS_GLOBAL_ORIGIN with HOME_LAT/LON/ALT as a backup EKF origin
    reset, in case ahrs:set_origin() inside Lua does not fully take effect.

OUTCOME: Both drones teleport back to home position between episodes. 🔄
         (Being verified — all root causes addressed)

KEY LESSON — pymavlink message handling:
  recv_match(type=X) reads messages from the TCP socket one at a time and
  DISCARDS any message that does not match the filter. Non-matching messages
  are NOT buffered anywhere — they are gone forever. This means:
  - While force_disarm() polls HEARTBEAT, all other messages are discarded.
  - Message-based synchronisation (polling PARAM_VALUE or STATUSTEXT for a
    trigger confirmation) is unreliable unless the code reads ALL message
    types and routes them itself (complex). Simple time.sleep() is more
    reliable for Lua synchronisation in this setup.

KEY LESSON — sim:set_pose() instance index:
  In multi-instance SITL, each ArduPilot process runs independently.
  The sim object in Lua refers to THAT INSTANCE'S simulator only.
  Index 0 always means "myself." Never use the SYSID-derived index.

KEY LESSON — Quaternion() in ArduPilot Lua:
  Quaternion() does not return an identity quaternion. Always call
  q:from_euler(roll, pitch, yaw) to initialise it before use.
